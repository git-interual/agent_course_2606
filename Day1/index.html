09:30

Transformer

원본 보기

핵심 장면

  • RNN은 이전 입력을 기억하지만 긴 문맥과 병렬 처리에 약하다.
  • LSTM은 기억 문제를 보완했지만 순차 처리 구조는 남아 있다.
  • Transformer는 Attention으로 문장 전체 관계를 한 번에 본다.
  • GPT는 Transformer의 Decoder Only 구조를 확장한 계열로 설명했다.

Transformer 구성

Input Embedding→Positional Encoding
Multi-Head Attention→Feed Forward
Masked Attention→다음 토큰 생성

Attention Is All You Need

강의에서는 RNN 없이 Attention만으로 자연어 처리가 가능하다는 전환점이 생성형 AI의 기반이 되었다고 정리했습니다.

산업 동향

GPU, 메모리, 전력, 냉각, 데이터센터가 AI 성장의 실물 인프라로 함께 부상한다는 흐름을 짚었습니다.

10:30

Azure & Token

원본 보기

Azure 실습 포인트

  • Azure Portal은 AWS Console과 유사한 웹 기반 관리 화면으로 설명.
  • 교육용 테넌트와 Microsoft Authenticator를 통해 MFA를 경험.
  • 리소스 그룹은 실습 단위와 비용 관리 단위로 이해.
  • Region 선택은 GPU 자원, 법률, 서비스 정책, 리스크를 함께 고려.

Tokenization 흐름

문장→Tokenization→Token Sequence
Transformer→다음 Token 예측→응답
정리: LLM은 글자 자체가 아니라 토큰을 처리합니다. 따라서 문장 길이, 비용, 응답 속도, 컨텍스트 크기는 모두 토큰 감각과 연결됩니다.

11:30

SLM & AI Ecosystem

원본 보기

Bigger is Better에서 Fit for Purpose로

무조건 큰 모델을 고르는 대신, 목적에 맞는 모델을 비용 효율적으로 선택하는 방향으로 산업의 관심이 이동하고 있습니다.

SLM

Small Language Model은 빠른 응답, 낮은 비용, 작은 메모리 사용량을 장점으로 하며 모바일, AI PC, 자동차, IoT에 연결됩니다.

MoE

Mixture of Experts는 모든 파라미터를 쓰지 않고 필요한 Expert만 활성화해 GPU 사용량과 추론 비용을 낮추는 접근입니다.

경량화

Distillation, Quantization, Pruning은 성능을 최대한 유지하면서 모델 크기와 추론 부담을 줄이는 대표 기법입니다.

On-device AI 장점

인터넷 없이 사용, 개인정보 보호, 빠른 응답, 저전력.

생태계 키워드

Ollama, Artificial Analysis, Qwen, Upstage, Hugging Face, GitHub.

13:30

Prompt Engineering

원본 보기

프롬프트의 4대 요소

Role+Context+Task+Format
  • Role은 AI의 역할과 말투를 바꾼다.
  • Context는 답변에 필요한 상황과 배경을 제공한다.
  • Task는 수행할 일을 분명히 한다.
  • Format은 결과물을 재사용 가능한 형태로 만든다.

System과 User

System = WHO

AI의 정체성, 우선 지침, 동작 경계.

User = WHAT

사용자가 요청하는 작업, 질문, 출력 요구.

Few-shot과 Fine-tuning

Fine-tuning은 비용과 시간이 크지만 모델 자체를 바꾸는 방법이고, Few-shot은 예시를 통해 빠르게 행동을 유도하는 방법입니다.

Structured Output

감정 분석처럼 결과를 JSON 등 구조화된 형식으로 요구하면 후속 프로그램에서 사용하기 쉬운 출력이 됩니다.

개발환경 / Azure OpenAI / AI Foundry

원본 보기

실습 경로

Azure Portal→Marketplace→Azure OpenAI
리소스 생성→AI Foundry→Model Catalog
모델 선택→배포→Playground

추론 시장

학습은 소수 기업이 제한적으로 수행하지만 추론은 전 세계 사용자가 매일 반복합니다. 그래서 비용, 속도, 안전성이 운영의 핵심이 됩니다.

Model Catalog

GPT, Phi, Llama, Gemma, Qwen, DeepSeek 등 다양한 모델을 목적과 비용에 맞춰 탐색하는 장소로 설명했습니다.

배포 선택 기준

Global Standard, Provisioned Throughput, TPM, Content Filter, RLHF 같은 운영 항목이 단순 모델 선택만큼 중요합니다.

16:00

Microsoft Foundry와 첫 번째 챗봇

원본 보기

Azure OpenAI 호출

  • Python에서 `openai.AzureOpenAI` 클라이언트를 생성.
  • API Key, Endpoint, API Version은 `.env`와 환경변수로 분리.
  • Azure AI Foundry의 deployment name을 `model` 파라미터로 사용.
  • Content Filter와 토큰 사용량 등 응답 구조를 확인.

System Prompt 효과

같은 질문이라도 System Prompt로 역할과 맥락을 지정하면 답변의 스타일과 연상 방향이 달라진다는 점을 실습했습니다.

System Prompt→역할 정의→응답 스타일 변화

chatbot.py

사용자 입력을 반복해서 받고 `q` 입력 시 종료하는 콘솔 챗봇입니다. `get_response()`가 Azure OpenAI 호출을 담당합니다.

poem.py

`주제,내용` 형식의 입력을 분리하고 temperature를 높여 창작형 시를 생성하는 AI 시인 예제입니다.

win.py

Tkinter GUI에서 입력에 쉼표가 있으면 AI 시인을, 없으면 일반 챗봇을 호출합니다. API 호출은 별도 스레드에서 처리합니다.

핵심 메시지

Playground는 실험실이고, Python 코드는 실제 서비스의 시작점입니다. 모델 배포 이후에는 애플리케이션 구조가 중요해집니다.

왜 Token을 사용하는가?

Character

장점

  • 모든 언어 표현 가능

단점

  • 토큰 수 증가
  • 연산 비용 증가
Word

장점

  • 의미 단위 처리

단점

  • 사전 크기 무한 증가
  • 신조어 처리 어려움
Subword(Token)

장점

  • 적당한 사전 크기
  • 신조어 처리 가능
  • 높은 압축 효율

강의 표현

Optimal Balance

GPT-5 한국어 토크나이저 개선

강의 언급

약 77% 개선

효과

  • 토큰 수 감소
  • 처리 효율 증가
  • 비용 절감
  • 한국어 이해 성능 향상

Token과 Weight

Token

입력 데이터 표현 방식

예)

I Love You
              ↓
              [40,13007,1608]
Weight

모델이 학습한 지식

예)

대한민국 → 서울
              고양이 → 야옹
              2 + 2 → 4

같은 관계가 Weight 안에 저장된다.

              Parameter 수
