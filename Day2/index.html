AutoGen + Semantic Kernel

Microsoft Agent Framework는 멀티 에이전트 오케스트레이션의 AutoGen과 Plugin, Memory, Function 중심의 Semantic Kernel 흐름을 통합한 방향으로 설명되었습니다.

LangChain은 생태계와 예제가 풍부하고, n8n은 Low-Code 자동화, LangGraph는 State와 Graph 기반 Workflow 설계에 초점을 둡니다.

Framework 비교

LangGraph가 처리 순서와 상태 그래프를 묻는다면, Microsoft Agent Framework는 누가 일을 할지, 어떤 Agent 팀으로 구성할지를 묻습니다.

10:30

Agent의 기억과 Tool Calling

원본 보기

LLM은 Stateless

  • LLM 단독으로는 이전 대화를 기억하지 않는다.
  • ChatGPT처럼 보이는 기억은 대화 이력을 다시 전달하기 때문에 가능하다.
  • Microsoft Agent Framework는 `AgentSession`으로 Multi-turn 상태를 관리한다.
  • 오래된 대화는 요약할 수 있지만 정보 손실과 열화가 생길 수 있다.

Memory 구조

InMemory

RAM에 저장하며 구현이 쉽지만 프로그램 종료 시 기록이 사라집니다.

Custom Provider

SQLite, PostgreSQL, Redis 등으로 영구 저장을 구성할 수 있습니다.

Tool 함수

`get_weather(location)` 함수에 `Annotated`와 `Field`로 파라미터 설명을 붙여 Agent가 도구의 목적과 입력값을 이해하도록 했습니다.

Agent에 Tool 등록

`OpenAIChatClient(...).as_agent(..., tools=[get_weather])` 형태로 Agent가 필요할 때 함수를 자동 선택할 수 있게 구성했습니다.

Tool 결과 신뢰

Tool이 이상한 값을 반환해도 LLM은 그 결과를 사실처럼 설명할 수 있습니다. 따라서 Agent 품질은 LLM 품질과 Tool 품질을 함께 요구합니다.

실습 코드

`add_tools.py`는 Azure OpenAI 기반 Agent에 날씨 도구를 연결하고 “프랑스 파리 날씨” 질문으로 Tool Calling을 확인합니다.

11:30

Session과 Multi-turn Conversation

원본 보기

Session이 필요한 이유

  • 각 `agent.run()` 호출은 기본적으로 독립적인 대화로 처리된다.
  • 이전 발화를 유지하려면 `agent.create_session()`으로 Session을 만든다.
  • 같은 Session을 넘기면 “내 이름이 뭐지?” 같은 후속 질문에 답할 수 있다.
  • 기억하는 것은 LLM 자체가 아니라 Session과 History 구조다.

Multi-turn 구조

User→Agent→Session→History
첫 질문→응답→후속 질문→기억 기반 응답

AI Debt

AI가 작성한 코드를 이해하지 못한 채 기능을 추가하면 유지보수, 버그 수정, 리팩토링이 어려워지는 새로운 기술 부채가 생깁니다.

Multi Tool Calling

`get_weather()`에 `get_exchange_money()`를 추가해 Agent가 날씨 조회와 환율 계산을 함께 수행하도록 구성했습니다.

도시와 통화 추론

사용자가 통화 코드를 말하지 않아도 Agent는 도쿄는 JPY, 뉴욕은 USD, 런던은 GBP처럼 도시와 통화를 연결해 Tool을 호출했습니다.

실습 코드

`multi_turn.py`는 Session 기반 대화 유지와 날씨, 환율 Tool을 함께 사용하는 복합 질문 실습을 담고 있습니다.
