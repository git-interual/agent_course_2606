Microsoft Agent Framework 입문

원본 보기

Agent 개발에서 자주 다루는 데이터

  • Markdown: 사람이 읽기 쉽고 LLM 친화적인 문서 형식.
  • JSON: API와 Agent 입출력에 자주 쓰이는 구조화 데이터.
  • CSV: 표 형태 데이터와 분석, Excel 호환에 유리한 형식.

Chatbot과 Agent

사용자→LLM→응답
사용자→Agent→Tool 선택→실행

AutoGen + Semantic Kernel

Microsoft Agent Framework는 멀티 에이전트 오케스트레이션의 AutoGen과 Plugin, Memory, Function 중심의 Semantic Kernel 흐름을 통합한 방향으로 설명되었습니다.

LangChain, n8n, LangGraph

LangChain은 생태계와 예제가 풍부하고, n8n은 Low-Code 자동화, LangGraph는 State와 Graph 기반 Workflow 설계에 초점을 둡니다.

Framework 비교

LangGraph가 처리 순서와 상태 그래프를 묻는다면, Microsoft Agent Framework는 누가 일을 할지, 어떤 Agent 팀으로 구성할지를 묻습니다.

hello_agent.py

`agent-framework` 설치 후 `OpenAIChatClient`, `AsyncAzureOpenAI`, `.env`, `asyncio.run()`으로 Agent 프로젝트의 기본 골격을 만들었습니다.

10:30

Agent의 기억과 Tool Calling

원본 보기

LLM은 Stateless

  • LLM 단독으로는 이전 대화를 기억하지 않는다.
  • ChatGPT처럼 보이는 기억은 대화 이력을 다시 전달하기 때문에 가능하다.
  • Microsoft Agent Framework는 `AgentSession`으로 Multi-turn 상태를 관리한다.
  • 오래된 대화는 요약할 수 있지만 정보 손실과 열화가 생길 수 있다.

Memory 구조

InMemory

RAM에 저장하며 구현이 쉽지만 프로그램 종료 시 기록이 사라집니다.

Custom Provider

SQLite, PostgreSQL, Redis 등으로 영구 저장을 구성할 수 있습니다.

Tool 함수

`get_weather(location)` 함수에 `Annotated`와 `Field`로 파라미터 설명을 붙여 Agent가 도구의 목적과 입력값을 이해하도록 했습니다.

Agent에 Tool 등록

`OpenAIChatClient(...).as_agent(..., tools=[get_weather])` 형태로 Agent가 필요할 때 함수를 자동 선택할 수 있게 구성했습니다.

Tool 결과 신뢰

Tool이 이상한 값을 반환해도 LLM은 그 결과를 사실처럼 설명할 수 있습니다. 따라서 Agent 품질은 LLM 품질과 Tool 품질을 함께 요구합니다.

실습 코드

`add_tools.py`는 Azure OpenAI 기반 Agent에 날씨 도구를 연결하고 “프랑스 파리 날씨” 질문으로 Tool Calling을 확인합니다.

Flow

Tool Calling 실행 흐름

사용자 질문→Agent→도구 선택
get_weather("파리")→도구 결과→최종 응답

11:30

Session과 Multi-turn Conversation

원본 보기

Session이 필요한 이유

  • 각 `agent.run()` 호출은 기본적으로 독립적인 대화로 처리된다.
  • 이전 발화를 유지하려면 `agent.create_session()`으로 Session을 만든다.
  • 같은 Session을 넘기면 “내 이름이 뭐지?” 같은 후속 질문에 답할 수 있다.
  • 기억하는 것은 LLM 자체가 아니라 Session과 History 구조다.

Multi-turn 구조

User→Agent→Session→History
첫 질문→응답→후속 질문→기억 기반 응답

AI Debt

AI가 작성한 코드를 이해하지 못한 채 기능을 추가하면 유지보수, 버그 수정, 리팩토링이 어려워지는 새로운 기술 부채가 생깁니다.

Multi Tool Calling

`get_weather()`에 `get_exchange_money()`를 추가해 Agent가 날씨 조회와 환율 계산을 함께 수행하도록 구성했습니다.

도시와 통화 추론

사용자가 통화 코드를 말하지 않아도 Agent는 도쿄는 JPY, 뉴욕은 USD, 런던은 GBP처럼 도시와 통화를 연결해 Tool을 호출했습니다.

실습 코드

`multi_turn.py`는 Session 기반 대화 유지와 날씨, 환율 Tool을 함께 사용하는 복합 질문 실습을 담고 있습니다.

13:30

MVP Tour Agent 만들기

원본 보기

여행 상담 Agent

  • `MVPTour-Assistant`라는 이름으로 여행 상담 역할을 부여했습니다.
  • Persona와 응답 규칙을 instructions에 넣어 상담원 말투를 고정했습니다.
  • 답변 끝에는 브랜드 문구를 반복하도록 Agent의 행동 규칙을 명시했습니다.

SDK 변화 대응

강의 자료의 `AzureOpenAIResponsesClient` 방식이 현재 설치 버전과 맞지 않아 `OpenAIChatClient`와 `AsyncAzureOpenAI` 조합으로 실습 코드를 조정했습니다.

Tool Decorator

`@tool(approval_mode="never_require")`로 날씨와 환율 함수를 Agent가 호출할 수 있는 공식 Tool로 선언했습니다.

Session 유지

Console Chatbot 구조에서 하나의 Session을 유지해 여행 상담 흐름과 후속 질문을 같은 맥락 안에서 처리했습니다.

Tool Calling 확인

도시별 날씨와 통화별 환율을 질문하며 Agent가 필요한 Tool을 선택하고 결과를 상담 문장으로 합치는 과정을 확인했습니다.

오늘 완성한 MVP

단순 응답 Agent에서 여행 상담 역할, 기억, 날씨 Tool, 환율 Tool을 갖춘 MVP Tour Agent로 확장했습니다.

14:30

RAG와 Vector Database

원본 보기

RAG가 필요한 이유

  • LLM 내부 지식만으로는 회사 정책이나 최신 문서를 정확히 답하기 어렵습니다.
  • Retrieval로 관련 문서를 찾고, Generation에서 그 근거를 활용합니다.
  • 검색은 키워드 중심, Retrieval은 의미와 맥락 중심으로 설명되었습니다.

Embedding과 ChromaDB

문장을 벡터로 바꿔 의미상 가까운 문서를 찾는 Embedding 개념을 배우고, 학습용 Vector Database로 ChromaDB를 사용했습니다.

AI 활용의 경제성

AI는 기술적으로 가능한 곳보다 대체 효과와 비용 절감이 큰 영역에서 먼저 확산된다는 관점으로 키오스크, 자율주행, 산업 자동화를 살폈습니다.

Ontology

Palantir 사례를 통해 데이터, 관계, 의미를 함께 정의하는 Ontology가 AI 기반 의사결정에서 어떤 역할을 하는지 정리했습니다.

MVP Tour RAG Tool

여행 상품과 정책 문서를 저장하고, Agent가 `search_travel_docs` 같은 RAG Tool을 호출해 상담 답변에 반영하는 구조를 구현했습니다.

관찰한 현상

첫 RAG 구현에서는 검색 품질, 문서 분할, Embedding, Tool 설명이 답변 품질을 함께 좌우한다는 점을 확인했습니다.

16:00

Real Function Calling과 Azure VM

원본 보기

Mock에서 Real로

  • 고정 배열과 사전으로 응답하던 Tool을 실제 외부 API 호출로 바꿨습니다.
  • Open-Meteo로 날씨를, ExchangeRate API로 환율을 조회했습니다.
  • LLM은 판단하고 Tool은 실제 시스템과 연결하는 구조를 확인했습니다.

Azure VM

Ubuntu Pro 24.04 LTS VM을 만들고, SLA와 가용성의 의미를 짚은 뒤 SSH로 접속해 서버 프로그램을 실행할 준비를 마쳤습니다.

SLA와 가용성

SLA는 장애가 없다는 약속이 아니라 약속한 가용성 수준을 넘겨 장애가 발생하면 보상하는 계약이라는 점을 정리했습니다.

Agent의 한계

Tool이 실제 API와 연결되어도 인증, 네트워크, 비용, 보안, 운영 환경까지 설계해야 실제 서비스로 이어질 수 있음을 다뤘습니다.

OpenClaw와 Discord

Azure VM 위에서 OpenClaw 설치를 시도하고, Telegram 대신 Discord 연동을 검토하며 Agent 운영 채널을 넓혔습니다.

교육 전체 흐름

LLM 호출에서 시작해 Agent, Tool, Session, RAG, 실제 API, VM 운영으로 확장되는 흐름을 한 줄로 연결했습니다.

from dotenv import load_dotenv
              import os
              import asyncio
              
              from agent_framework.openai import OpenAIChatClient
              from openai import AsyncAzureOpenAI
              
              from pydantic import Field
              from typing import Annotated
              
              load_dotenv()
              
              
              def get_weather(
                  location: Annotated[
                      str,
                      Field(
                          description=
                          "도시 이름. 서울, 도쿄, 파리 등"
                      )
                  ]
              ) -> str:
              
                  conditions = [
                      "맑음",
                      "흐림",
                      "비",
                      "눈"
                  ]
              
                  print(
                      f"Tool get_weather called with location: "
                      f"{location}"
                  )
              
                  return (
                      f"{location}의 오늘 날씨는 "
                      f"{conditions[0]}입니다."
                  )
              
              
              async def main() -> None:
              
                  endpoint = os.getenv(
                      "AZURE_OPENAI_ENDPOINT"
                  )
              
                  api_key = os.getenv(
                      "AZURE_OPENAI_API_KEY"
                  )
              
                  model = os.getenv(
                      "AZURE_OPENAI_CHAT_MODEL"
                  )
              
                  api_version = os.getenv(
                      "AZURE_OPENAI_API_VERSION"
                  )
              
                  azure_client = AsyncAzureOpenAI(
                      azure_endpoint=endpoint,
                      api_key=api_key,
                      api_version=api_version,
                  )
              
                  agent = OpenAIChatClient(
                      async_client=azure_client,
                      model=model,
                  ).as_agent(
                      name="HelloAgent",
                      instructions=
                          "너는 친절한 AI 비서야.",
                      tools=[
                          get_weather
                      ],
                  )
              
                  result = await agent.run(
                      "프랑스 파리 날씨는 어때?"
                  )
              
                  print(result)
              
              
              if __name__ == "__main__":
                  asyncio.run(main())
